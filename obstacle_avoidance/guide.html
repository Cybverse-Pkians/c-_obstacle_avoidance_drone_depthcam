What Goes Where — Complete File Map

Every single file, its exact path on each machine, and what it does. Read this once and you'll never be confused again.

FileLaptop pathPi pathPurpose
package.xml ~/drone_project/ros2_pkg/ ~/ros2_ws/src/drone_avoidance/ Declares ROS2 dependencies
CMakeLists.txt ~/drone_project/ros2_pkg/ ~/ros2_ws/src/drone_avoidance/ Build instructions for colcon

Never edit files directly on the Pi. Always edit on the laptop, then run deploy.sh. This keeps laptop as your single source of truth.
Never touch build/ or install/ folders. colcon creates and manages these automatically. Deleting and recreating them is fine — run colcon build again and they come back.

#!/bin/bash
# Run this from laptop whenever you change any file
# Usage: ./deploy.sh

PI="pi@10.42.0.1"
PROJECT="$HOME/drone_project"

echo "📦 Copying files to Pi..."

scp -r "$PROJECT/ros2_pkg/." "$PI:~/ros2_ws/src/drone_avoidance/"

  cd ~/ros2_ws && \
  colcon build --symlink-install --packages-select drone_avoidance && \
  source install/setup.bash && \
  echo '✅ Build complete'"

echo "🌐 GUI at: http://10.42.0.1:8080/drone_gui.html"

# Make it executable — run once

Ubuntu Laptop · Step 3

Connect to Pi and Deploy

Set up SSH access from laptop to Pi. Do this once, then use deploy.sh every time you change a file.

1
Connect laptop to Pi's WiFi hotspot
Pi creates a hotspot called DroneNet. Connect to it from Ubuntu's WiFi menu. Pi's IP is always 10.42.0.1.
BASH💻 Laptop terminal
# Test connection
ping -c 3 10.42.0.1
2
Set up passwordless SSH (do once)
So deploy.sh doesn't ask for password every time.
BASH💻 Laptop terminal
# Generate SSH key if you don't have one
ssh-keygen -t ed25519 -C "drone_laptop"
# Press Enter 3 times (accept defaults, no passphrase)

# Copy key to Pi
ssh-copy-id pi@10.42.0.1
# Enter Pi password once — never again after this
3
Run deploy.sh — pushes everything to Pi and builds
BASH💻 Laptop terminal
cd ~/drone_project && ./deploy.sh
After this you're done on the laptop. Everything is now on the Pi.

Raspberry Pi · Step 5

Enable UART for Cube+ Connection

The Pi talks to the Cube+ over its hardware UART pins. This enables that port and disables the default login shell that's using it.

1
Run raspi-config
BASH🥧 Pi terminal
sudo raspi-config
Navigate: Interface Options → Serial Port
"Would you like a login shell to be accessible over serial?" → No
"Would you like the serial port hardware to be enabled?" → Yes
Finish → Reboot
2
After reboot — verify UART exists
BASH🥧 Pi terminal
ls /dev/serial0
# Must print: /dev/serial0
# If it says "No such file" — redo raspi-config step
3
Wire Cube+ TELEM2 to Pi GPIO
Cube+ TELEM2 pinSignalPi GPIO pinPi physical pin
Pin 2VCC 5V5VPhysical pin 4
Pin 3GNDGNDPhysical pin 6
Pin 4TXGPIO15 (RXD)Physical pin 10
Pin 5RXGPIO14 (TXD)Physical pin 8
TX goes to RX and RX goes to TX — they cross over. This is correct and intentional.
Raspberry Pi · Step 6

Build the ROS 2 Package on Pi

Building happens on the Pi, never on the laptop. You always build after deploy.sh copies the files. The deploy.sh script does this automatically, but here's how to do it manually.

1
Speed up build — use all 4 Pi cores
Create colcon.meta once at the workspace root. Reduces build time from 12 min → 3 min.
BASH🥧 Pi terminal
cd ~/ros2_ws
cat > colcon.meta << 'EOF'
{"names":{"colcon-core":{"args":["--parallel-workers","4"]}}}
EOF
2
Build the package
Run from ~/ros2_ws. Run this every time deploy.sh copies new .cpp files.
BASH🥧 Pi terminal — ~/ros2_ws
cd ~/ros2_ws
colcon build --symlink-install --packages-select drone_avoidance
--symlink-install means launch files (.py) and config (.yaml) are symlinked — change them and just re-launch, no rebuild needed.
--packages-select drone_avoidance only builds your package, skips everything else.
3
Verify build succeeded
BASH🥧 Pi terminal
ros2 pkg executables drone_avoidance
# Must show:
# drone_avoidance fake_depth_node
# drone_avoidance side_obstacle_avoidance
If it shows nothing or errors — the build failed. Scroll up in the build output to find the C++ error. Fix the .cpp file on the laptop, run deploy.sh again, rebuild.
Raspberry Pi · Step 7

Sourcing — The Most Important Thing

This is where most beginners get confused. You need to "source" two files in every terminal before using ROS 2. Think of it like activating ROS 2 for that terminal session.

📦 What sourcing does
It adds ROS 2 commands and your package's executables to the terminal's PATH. Without sourcing, ros2 launch drone_avoidance ... says "package not found".
⚠ Must source EVERY terminal
Each new SSH session or terminal tab is a fresh shell — sourcing from one terminal does NOT carry over to another. Open new tab → source again.
!
Make sourcing automatic — add to ~/.bashrc
Do this once on the Pi. After this, every new terminal automatically sources both files.
BASH🥧 Pi terminal
# Add these 4 lines to ~/.bashrc
echo 'source /opt/ros/humble/setup.bash' >> ~/.bashrc
echo 'source ~/ros2_ws/install/setup.bash' >> ~/.bashrc
echo 'export ROS_DOMAIN_ID=42' >> ~/.bashrc
echo 'export ROS_LOCALHOST_ONLY=0' >> ~/.bashrc

# Apply immediately to current terminal
source ~/.bashrc
!
Do the same on your Ubuntu laptop
So your laptop can see Pi's ROS 2 topics over the network.
BASH💻 Laptop terminal
echo 'source /opt/ros/humble/setup.bash' >> ~/.bashrc
echo 'export ROS_DOMAIN_ID=42' >> ~/.bashrc
echo 'export ROS_LOCALHOST_ONLY=0' >> ~/.bashrc
source ~/.bashrc
Both Pi and laptop must have the same ROS_DOMAIN_ID=42. This is how they find each other's topics across the WiFi network.
!
What the two source lines mean
CommandWhat it sourcesWhen needed
source /opt/ros/humble/setup.bash ROS 2 Humble core — ros2 command, all standard packages Every terminal, always first
source ~/ros2_ws/install/setup.bash Your drone_avoidance package — ros2 run/launch drone_avoidance ... After every colcon build
The second source only works after you've run colcon build at least once. If install/ doesn't exist yet, it will error — that's normal, just build first.
Raspberry Pi · Step 8

Serve the GUI — Auto-start on Boot

Set up the Pi to serve drone_gui.html on port 8080 automatically every time it powers on. Access it from any device on the DroneNet hotspot.

1
Create a systemd service (runs on boot, forever)
BASH🥧 Pi terminal
sudo tee /etc/systemd/system/drone-gui.service > /dev/null << 'EOF'
[Unit]
Description=Drone Control GUI Server
After=network.target

[Service]
Type=simple
User=pi
WorkingDirectory=/home/pi/drone_gui
ExecStart=python3 -m http.server 8080
Restart=always
RestartSec=3

[Install]
WantedBy=multi-user.target
EOF
2
Enable and start it
BASH🥧 Pi terminal
sudo systemctl daemon-reload
sudo systemctl enable drone-gui
sudo systemctl start drone-gui

# Check it's running
sudo systemctl status drone-gui
# Should show: Active: active (running)
3
Open the GUI
From any device connected to DroneNet hotspot — laptop, phone, tablet:
http://10.42.0.1:8080/drone_gui.html
The GUI is now permanently available whenever the Pi is powered on and you're connected to DroneNet. No SSH needed to open it.
Every Session

Exact Launch Sequence — Every Bench Test

Follow this exact order every time. Open 3 separate SSH terminals on the Pi (or 3 tabs in the same SSH session).

Remove propellers before every bench test. Motors arm immediately when avoidance node starts.
0
Open the GUI in your browser first
BROWSER💻 Laptop or phone
http://10.42.0.1:8080/drone_gui.html
1
SSH Terminal 1 — Start MAVROS
Always first. Wait until you see "FCU connected" in the output before opening terminal 2.
BASH🥧 Pi — SSH terminal 1
ros2 launch drone_avoidance mavros_cube.launch.py
# Wait for: [ INFO] FCU: APM: ArduCopter ... connected
2
SSH Terminal 2 — Start bench test (fake depth + avoidance)
This is the "press Enter" moment. Starts both fake_depth_node and side_obstacle_avoidance together.
BASH🥧 Pi — SSH terminal 2
ros2 launch drone_avoidance bench_test.launch.py
# Watch: WARMUP → GUIDED → ARMING → TAKEOFF → MISSION → CRUISING
3
SSH Terminal 3 — Inject obstacles (or use GUI)
BASH🥧 Pi — SSH terminal 3
# Front obstacle — triggers LATERAL_SHIFT
ros2 param set /fake_depth_node front_dist 2.5

# Left wall 10 seconds — tests WALL_FOLLOWING
ros2 param set /fake_depth_node left_dist 3.0
ros2 param set /fake_depth_node left_duration 10.0

# Slanted right wall — tests slant detection
ros2 param set /fake_depth_node right_dist 2.0
ros2 param set /fake_depth_node right_slant true

# Clear all
ros2 param set /fake_depth_node front_dist -1.0
ros2 param set /fake_depth_node left_dist  -1.0
ros2 param set /fake_depth_node right_dist -1.0
4
Monitor from laptop (optional)
BASH💻 Laptop terminal
# Watch Cube+ connection state
ros2 topic echo /mavros/state

# Watch velocity commands being sent
ros2 topic echo /mavros/setpoint_velocity/cmd_vel_unstamped

# See all running nodes
ros2 node list
!
Emergency stop
BASH🥧 Pi — any terminal
ros2 service call /mavros/cmd/arming \
  mavros_msgs/srv/CommandBool "{value: false}"
One-time · Mission Planner

ArduPilot Parameters

Set these once in Mission Planner via your SiK radio. Connect as normal → Config → Full Parameter List → search each param → set → Write Params.

ParameterValueWhy
SERIAL2_PROTOCOL2MAVLink 2 on TELEM2 (Pi UART port)
SERIAL2_BAUD921921600 baud — matches mavros_cube.launch.py
SERIAL1_PROTOCOL2MAVLink on TELEM1 (your SiK radio, unchanged)
SERIAL1_BAUD5757600 baud — standard SiK radio speed
GUID_OPTIONS0Allow velocity commands in GUIDED mode
FS_THR_ENABLE1Throttle failsafe — safety net
DISARM_DELAY0Don't auto-disarm during bench test
SERIAL1 (SiK radio) and SERIAL2 (Pi UART) run simultaneously and independently. Your Mission Planner telemetry is unaffected.
